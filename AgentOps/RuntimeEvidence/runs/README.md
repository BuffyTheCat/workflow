# Runtime Runs

Each task that warrants persistent run-state gets its own subdirectory under this folder.

## Naming

```
AgentOps/RuntimeEvidence/runs/<YYYY-MM-DD>-<short-slug>/
```

Slug examples:

- `2026-05-02-sr-34787-signature-date`
- `2026-05-02-investigation-auth-flow`
- `2026-05-02-pr-34988-review`

Slugs are lowercase, hyphenated, ticket-id-first when applicable.

## Tier-based artifact requirements

These are minimum artifact requirements. Per canon §22 ("сначала ядро, потом расширять") and §17.3 (RUN_STATE max 1–2 screens, details elsewhere), the goal is **evidence as artifacts, not artifacts as evidence**. Produce only the artifacts whose content was actually generated by the run; do not pre-emptively create empty templates to satisfy a checklist.

| Tier | Mandatory minimum | Produced when phase actually happens | Never required by tier alone |
|---|---|---|---|
| T0 | none (inline only — typo, copy, obvious local fix) | — | n/a |
| T1 | `FINAL_REPORT.md` | `RUN_STATE.md` if resumable; `EVIDENCE_LEDGER.md` if more than one cited fact | LIVE_STATUS, HYPOTHESES, MEDIA_PACKET, SKEPTIC_REVIEW unless triggered |
| T2 | `LIVE_STATUS.md`, `EVIDENCE_LEDGER.md`, `SKEPTIC_REVIEW.md`, `FINAL_REPORT.md` | `RUN_STATE.md` (if resumable / before compaction), `TASK_CARD.md` (if ticket present), `TICKET_PACKET.md` (if MCP results), `MEDIA_PACKET.md` (if media items present), `HYPOTHESES.md` (if multiple competing causes), `HISTORY_PACKET.md` (if regression / prior fix), `PLAN.md` (only when Skeptic verdict allows, see `Core/SKEPTIC_PROTOCOL.md` Verdict Gating Rules) | n/a |
| T3 | T2 mandatory set + `RED_TEAM_REVIEW.md` (if security/auth/billing/data-loss/migration/tenant trigger) + `VERIFICATION.md` (if writes happened) | `PLAN.md` (per Skeptic gating), `DIFF_SUMMARY.md` (if writes), `LESSONS_CANDIDATE.md` (if reusable lesson) | n/a |
| T4 | `EVIDENCE_LEDGER.md`, `FINAL_REPORT.md` (read-only options/tradeoff matrix; recommendation, not implementation) | `TASK_CARD.md`, `PLAN.md` (read-only options) | implementation artifacts |

T0 tasks produce no run directory. Their output is the inline chat final summary.

There is no "no opt-out" hard-case escalation rule. Hard-case T2/T3 triggers (reopen, multi-comment, parent/related issues, frontend/backend split, runtime/media closure dependency, branch/build divergence — see `Core/AGENT_OS.md`) require the run to **actually do** the deeper investigation (more body reads, more media inspection, dedicated Skeptic / Contrarian per `Core/SKEPTIC_PROTOCOL.md`). Producing a longer artifact list is not a substitute. If the deeper investigation does not happen, the run is `REVISE_REQUIRED` per Skeptic Verdict Gating Rules — not `READY_FOR_IMPLEMENTATION_PLAN` with more files attached.

`PLAN.md` is special: it may be created only when Skeptic verdict permits (`PASS`, `PASS_WITH_RISKS`, `READY_TO_MUTATE`). Under `REVISE_REQUIRED` or `BLOCKED`, `PLAN.md` may contain only blockers, fastest falsifiers, next-evidence reads, and the explicit statement that no plan is available — see `Core/SKEPTIC_PROTOCOL.md` Verdict Gating Rules.

`LIVE_STATUS.md` is mandatory for T2+ because `Core/AGENT_OS.md` uses it as the pre-mutation Agent Dispatch Gate surface. Progress beacons may still be compact, but the gate must be observable there before the first code mutation or first `implementation_target:` marker. A gate reconstructed later in `FINAL_REPORT.md` is post-hoc by default.

## Evidence manifest (`EVIDENCE_MANIFEST.tsv`) and snippet contract

Optional in general; **required** when the run uses an `implementation_target:` marker in `FINAL_REPORT.md` or `PLAN.md`. Tab-separated, **five columns** (six with optional `commit_ref`):

```
path<TAB>status<TAB>lines<TAB>evidence_ref<TAB>snippet_hash[<TAB>commit_ref]
```

Example (5-column working-tree binding):

```
react/src/apps/ov/components/e-sign-viewer/eSignInputStyles.ts<TAB>BODY_READ<TAB>22-24<TAB>EVIDENCE_LEDGER.md<TAB>4997c301998e1fc78448f02292ca3d61860fe53e56d601e56323b0764fc6113c
```

Example (6-column commit-snapshot binding):

```
react/src/apps/ov/components/e-sign-viewer/eSignInputStyles.ts<TAB>BODY_READ<TAB>22-24<TAB>EVIDENCE_LEDGER.md<TAB>4997c301998e1fc78448f02292ca3d61860fe53e56d601e56323b0764fc6113c<TAB>d399e65a93
```

Column meanings:

- `path` — repo-relative file path. Must NOT be absolute (no leading `/`) and must NOT contain `..` segments. Path-traversal and absolute paths are rejected by the validator before any file read, to prevent the manifest from being used to claim reads of arbitrary files outside the target repo.
- `status` — one of `BODY_READ` / `DISCOVERY_ONLY` / `MEDIA_READ` / `MEDIA_METADATA_ONLY` / `BLOCKED_TOOL` / `BLOCKED_AUTH` / `BLOCKED_EXPIRED_URL` / `NEXT_READ`.
- `lines` — concrete line range (`22-24`) or single line (`41`). **Must NOT be** `unknown` / `n/a` / `?` / `-` / empty.
- `evidence_ref` — pointer to the in-run evidence artifact that carries the snippet block, of the form `FILENAME` or `FILENAME:LINE`.
- `snippet_hash` — sha256 of the **normalized** excerpt (see normalization rules below). Must equal the value in the `BODY_READ_SNIPPET:` block, the value the validator computes from the pasted excerpt, AND the value the validator computes from the actual file content at `<repo-root>/<path>` lines `<lines>` (or `git show <commit_ref>:<path>` if column 6 is set).
- `commit_ref` (optional, 6th column) — git commit, tag, or ref to validate against. When set, the validator runs `git -C <repo-root> show <commit_ref>:<path>` and binds the snippet to that commit's content. When unset, the validator binds the snippet to the working-tree file at `<repo-root>/<path>`.

The referenced evidence file must contain a structured snippet block:

```
BODY_READ_SNIPPET:
path: <same path as manifest>
lines: <same lines as manifest>
snippet_hash: <same snippet_hash as manifest>
excerpt:
<actual code excerpt from those lines>
END_BODY_READ_SNIPPET
```

For every `implementation_target: <path>` declared in `FINAL_REPORT.md` or `PLAN.md`, the validator enforces all of:

1. `<path>` appears in the manifest with `status=BODY_READ`.
2. `lines` is concrete (not `unknown`/`n/a`/`?`/`-`/empty).
3. `evidence_ref` is non-empty and the referenced file exists in the run directory.
4. The referenced file contains `<path>` and the `lines` string as literal substrings.
5. The referenced file contains a `BODY_READ_SNIPPET:`...`END_BODY_READ_SNIPPET` block whose `path:` value matches `<path>`.
6. The block's `path` / `lines` / `snippet_hash` all match the manifest row.
7. The block's excerpt is non-empty, ≥10 non-whitespace chars, not placeholder text (`code was read here` / `placeholder` / `the agent read` / etc.), and contains code-like syntax appropriate to the file extension.
8. The manifest `snippet_hash` equals the validator-computed sha256 of the **normalized** excerpt.
9. The `lines` field is a strict numeric range `N-M` with `N <= M`.
10. The actual repo content at `<repo-root>/<path>` lines `N..M` (or `git show <commit_ref>:<path>` lines `N..M` if a 6th `commit_ref` column is set), normalized identically to the excerpt, hashes to the same value as `snippet_hash` AND its normalized text equals the pasted excerpt byte-for-byte. **This is file-snapshot binding** — the manifest claim is bound not only to the pasted excerpt but to the actual file content. Adversarial fixtures with internally-consistent manifest/snippet hashes but fabricated excerpts now fail at this rule.

Rule 10 requires a target repository root. Pass it explicitly:

```
scripts/close-run.sh <run-dir> --repo-root /absolute/path/to/target/repo
```

Or via environment fallback:

```
AGENTOPS_TARGET_REPO=/absolute/path/to/target/repo scripts/close-run.sh <run-dir>
```

Runs **without** any `implementation_target:` marker do not require `--repo-root` — the file-snapshot phase only fires when an actionable target is declared.

Runs **with** an `implementation_target:` and **no** `commit_ref` and **no** `--repo-root` fail with the canonical error:

```
implementation_target '<path>' has no commit_ref and no --repo-root was supplied;
cannot validate against actual file. Pass --repo-root <target-repo-path> to close-run.sh
```

Runs with `commit_ref` set on every row may still need `--repo-root` because the validator invokes `git -C <repo-root> show <commit_ref>:<path>`. The error in that case names `commit_ref` explicitly so the operator can choose between (a) supplying repo-root and using commit-snapshot binding, or (b) dropping `commit_ref` to use working-tree binding.

### Excerpt normalization (canonical, for hash computation)

The validator computes the hash of the excerpt as follows. An agent generating a manifest entry must reproduce this exactly:

1. Take the lines printed between `excerpt:` and `END_BODY_READ_SNIPPET` (exclusive on both ends).
2. Trim **exactly one** leading blank line if the first captured line is blank/whitespace-only.
3. Trim **exactly one** trailing blank line if the last captured line is blank/whitespace-only.
4. Preserve every other line's content as-is, including indentation.
5. Join the resulting lines with `\n` between them. **No trailing `\n` after the final line.**
6. Compute `sha256` of the resulting bytes.

Equivalent shell (assuming `$EXCERPT` is the multi-line excerpt content with no trailing newline):

```sh
printf '%s' "$EXCERPT" | shasum -a 256 | awk '{print $1}'
```

For a multi-line snippet pulled from a file with `sed -n '22,24p' path/to/file`, drop the trailing newline (`sed` adds one to each line) before hashing, or use `printf '%s'` from a captured-and-stripped variable.

What this does and does not prove:

- **Does** prevent single-artifact laundering. An agent must produce a manifest row, a referenced evidence file, a structured snippet block, and a code-like excerpt — all four mutually consistent on path / lines / hash. Single-row laundering (`path<TAB>BODY_READ`) is rejected.
- **Does** bind the manifest claim to actual repository content (file-snapshot or commit-snapshot binding). An adversary who fabricates an excerpt and computes its hash now fails at rule 10 unless the fabricated excerpt happens to match the actual file content at the cited lines — which collapses the attack to "tell the truth". The diagnostic message names the source: `working-tree=<repo-root>` or `commit_ref=<sha>`.
- **Does NOT** prove the Read tool was invoked. The contract binds: manifest ↔ pasted excerpt ↔ actual file content. It does not bind: actual file content ↔ tool invocation log. An agent that read the file via `cat` outside the Read tool, or that copy-pasted from a colleague's terminal, can still produce a passing manifest. Tool-bound evidence (binding claims to actual Read tool output logs) would require instrumenting the agent runtime — out of scope. The current contract is `FILE_SNAPSHOT_BOUND_BUT_NOT_TOOL_LOG_BOUND` (working-tree) or `COMMIT_SNAPSHOT_BOUND_BUT_NOT_TOOL_LOG_BOUND` (commit_ref).

Runs that produced an `implementation_target:` claim before this contract landed are conceptually labeled `PRE_SNIPPET_CONTRACT_RUN` (no automated marker; the convention is to disclose the run-date or refer to this README from a `RUN_STATE.md`). They are not retro-edited to satisfy the contract; they are evidence of prior behavior.

## Structured markers in `FINAL_REPORT.md` / `PLAN.md`

To make actionable claims auditable, use these literal markers (one per line, anywhere in the doc):

- `implementation_target: <path>` — declares an actionable fix target. Forbidden under Skeptic `REVISE_REQUIRED`/`BLOCKED`. Path must be in manifest as `BODY_READ`.
- `next_read: <path>` — declares a file the next agent must body-read. Allowed under any verdict. Must be disjoint from `implementation_target:` paths.
- `blocked_media: <id>` — declares a media item that blocks closure.
- `required_falsifier: <description>` — declares a concrete next-evidence read or test that would unblock.

Prose that combines a mutation verb (`change`, `modify`, `update`, `fix`, `edit`, `patch`, `implement in`, `route through`, `target`, `minimal change in`) with a code-extension path (`.ts`, `.tsx`, `.cs`, `.py`, etc.) is forbidden under Skeptic `REVISE_REQUIRED`/`BLOCKED`/`STOP_BEFORE_WRITE` and under any non-implementation executive verdict. Use `next_read: <path>` or `required_falsifier: <description>` markers instead. Lines explicitly framed as historical citation (containing `prior fix`, `previous fix`, `FACT_GIT`, `FACT_CODE`, `git log/show/blame/diff`, etc.) are excluded — descriptive evidence about past commits is not directive advice.

`.md` paths (workflow artifacts like `RUN_STATE.md`, `EVIDENCE_LEDGER.md`) are excluded from the actionable-prose check by extension. Operators / downstream tools should still prefer reading the structured markers, not the prose, when acting on the run.

## Closing a run (`MANUAL_CLOSE_CHECK`)

T2+ runs SHOULD pass this command before closure can be claimed:

```
scripts/close-run.sh <run-dir> [--repo-root <absolute-path-to-target-repo>]
```

The `--repo-root` flag is required only when the run carries an `implementation_target:` marker (see "Evidence manifest" above). Runs without that marker do not need it. `AGENTOPS_TARGET_REPO=<path>` is honored as an environment fallback.

`close-run.sh` wraps `validate-run-report.sh`. It prints `CLOSE_OK` on pass or `CLOSE_BLOCKED` on fail and exits non-zero. The validator checks Verdict Gating Rules, Inspection-budget verdict cap, the structured-marker / manifest relation, the file-snapshot (or commit-snapshot) binding when an implementation target is declared, and the executive-verdict allow-list.

Hook-based enforcement is **not implemented** — do not claim hook enforcement, and do not call this check "mandatory" in any mechanical sense. It is a manual-invocation check. An agent that produces `SKEPTIC_REVIEW.md` and `FINAL_REPORT.md` and stops without invoking `close-run.sh` leaves no disk evidence that the check was skipped (the script does not currently write a persistent marker). Future operators auditing the run dir cannot distinguish "ran and passed" from "never ran" without external telemetry. The right invocation surface for a Stop hook depends on consuming-project context (which run dir is "active") that portable AgentOps source cannot decide alone; if a consuming project wants enforcement, it lives in that project's `.claude/settings.json`, not in AgentOps.

A run that *fails* `close-run.sh` (when the agent invokes it and the validator returns non-zero) is `INVALID_WORKFLOW_RUN` regardless of narrative. A run that *skips* `close-run.sh` is unverified-by-machine; treat its claims as self-attested.

## Persistence rules

- Do not commit `runs/<...>/` content. The directory is gitignored in Inline mode and lives under `.agentops-local/` in Overlay mode (see `Core/DEPLOYMENT_MODES.md`).
- Do not store secrets, tokens, PII, full env dumps, or full chat transcripts. Use redacted summaries per `Core/COMMIT_PUSH_POLICY.md` and the redaction conventions in `Core/CONTEXT_COMPACTION_POLICY.md`.
- A run directory is closed when its `FINAL_REPORT.md` is filed. Older closed runs are subject to retention policy: keep all for 30 days, then keep only `FINAL_REPORT.md` + `LESSONS_CANDIDATE.md` and remove the rest after 90 days unless marked KEEP.
